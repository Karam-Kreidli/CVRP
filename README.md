# CVRP Repository

This repository contains an RL-guided CVRP system built around the GECCO ML4VRP workflow:

- A trainable policy chooses solver strategies.
- HGS-CVRP performs the actual route optimization.
- Baseline scripts provide reproducible non-RL comparisons.
- A backend service exposes solve and benchmark APIs.

The optimization objective used throughout the codebase is:

`Score = 1000 * NV + TD` (lower is better)

- `NV`: number of vehicles.
- `TD`: total route distance.

The `1000 * NV` term makes fleet size reduction the dominant objective.

## Why This Repository Exists

This project exists to answer one practical question:

Can a lightweight RL policy improve how a strong deterministic CVRP solver (HGS) is steered over repeated solve steps?

Instead of replacing HGS, the policy decides *how* HGS should search at each step:

- free fleet vs constrained fleet,
- same seed vs new seed,
- standard vs stronger iteration budgets,
- default vs action-specific HGS search biases.

## End-to-End Running Mechanism

At a high level, each solve/training episode follows this flow:

1. Load one `.vrp` instance from `Data/`.
2. Compute 12 hand-crafted instance features.
3. Run an initial HGS solve (baseline candidate).
4. For up to 50 steps:
   - Policy reads current observation (`12 instance features + 7 solver stats`),
   - Policy picks one of 10 actions,
   - Environment maps action to HGS parameters and runs a fresh solve,
   - Environment updates best-so-far solution and emits reward.
5. During training, PPO updates the policy from collected rollouts.
6. During inference/evaluation, greedy actions are used and artifacts are exported.

## Folder Purpose Map (Why Each Exists)

- `Backend/`: online API service for solve/status/result/benchmark workflows.
- `Baseline/`: offline benchmarking and comparison scripts (RL-vs-baseline and pure HGS baselines).
- `Model/`: core RL + environment implementation (policy, environment, PPO trainer, CLI entrypoint).
- `Data/`: source CVRP instances used for training and evaluation.
- `Documents/`: architecture, code walkthrough, and literature benchmark references.
- `Logs/`: persistent training run artifacts and best model file.
- `Checkpoints/`: resumable snapshots from periodic and final training saves.
- `Results/`: analysis/report artifacts generated by comparison runs.
- `Solutions/`: exported route files in `.sol` and `.txt` competition-style formats.
- `CVRP-ENV/`: local Python environment snapshot for reproducible local execution.
- `.vscode/`: workspace editor settings.
- `__pycache__/`: interpreter-generated bytecode cache.

## Setup

## 1) Python Environment

You can either use the existing local environment folder (`CVRP-ENV/`) or create your own.

Example using a fresh virtual environment:

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install --upgrade pip
pip install -r requirements.txt
```

## 2) Data Availability Check

```bash
ls Data/*.vrp | wc -l
```

Expected in current repo snapshot: `59`.

## Main Workflows

## A) Run Smoke Tests (Model + Env + PPO mini loop)

```bash
python -m Model.main
```

This verifies:

- policy forward/action sampling,
- environment stepping and masking,
- short PPO training loop,
- checkpoint save/load basics.

## B) Train Policy

```bash
python -m Model.main train \
  --instance_path Data \
  --epochs 50 \
  --episodes_per_epoch 8 \
  --save_interval 10 \
  --checkpoint_dir Checkpoints \
  --log_dir Logs
```

Common optional controls:

- `--fp16`
- `--eval_count`
- `--holdout_count`
- `--holdout_eval_interval`
- `--best_model_metric {eval,holdout,composite}`
- `--resume Checkpoints/Checkpoint_EpochXX.pth`
- `--start_epoch N`

## C) RL Inference (No Baseline)

```bash
python Baseline/Infer.py \
  --instance_dir Data \
  --model_path Logs/Best_Model.pth \
  --solution_output_dir Solutions/HGS+RL
```

## D) RL vs Baseline Comparison + Reports

```bash
python Baseline/Infer.py \
  --instance_dir Data \
  --model_path Logs/Best_Model.pth \
  --baseline \
  --baseline_iters 25000 \
  --baseline_cache_csv Results/Competition_Eval_Baseline_Cache.csv \
  --report_dir Results \
  --report_stem Competition_Eval \
  --solution_output_dir Solutions/HGS+RL
```

Produces:

- `Results/<stem>_Instances.csv`
- `Results/<stem>_Summary.json`
- `Results/<stem>_Report.md`
- `Solutions/HGS+RL/sol-Format/*.sol`
- `Solutions/HGS+RL/txt-Format/*.txt`

## E) HGS Baseline Evaluation (Default + Large Population)

```bash
python Baseline/Baseline_Evaluation.py \
  --instance_path Data \
  --nb_iter 25000
```

## F) Portfolio Baseline (Strong Non-RL Baseline)

```bash
python Baseline/Portfolio_Solver.py \
  --instance_path Data \
  --nb_iter 10000 \
  --num_seeds 5 \
  --workers 4 \
  --output_dir Solutions/HGS
```

## G) Start API Backend

```bash
python Backend/Backend.py
```

Then open FastAPI docs in browser:

- `http://localhost:8080/docs`

## Artifacts and Their Roles

- `Logs/Best_Model.pth`: best checkpoint chosen by configured selection metric.
- `Logs/Training_Metrics.csv`: epoch-level training metrics.
- `Logs/Run_Configuration.json`: captured run settings for reproducibility.
- `Checkpoints/Checkpoint_Epoch*.pth`: periodic resume points.
- `Checkpoints/Checkpoint_Final.pth`: final snapshot of run state.
- `Results/*`: analysis-friendly summary outputs.
- `Solutions/*`: route exports intended for manual inspection and downstream use.

## Reproducibility Notes

- Randomness appears in instance sampling, action sampling during training, and some solver seed paths.
- Inference scripts support explicit seeds and baseline seed lists/policies.
- Baseline cache in `Baseline/Infer.py` uses a config signature to avoid mixing incompatible settings.

## Known Operational Notes

- Backend job state is in-memory only; restarting the service clears active/completed run state.
- `CVRP-ENV/` is machine-local environment state, not algorithm source code.
- `Baseline/Setup_VM.sh` is a helper script and contains legacy path strings in comments/commands; treat it as a starting template rather than strict source of truth.
